### **1. OpenAI Launches Codex Security: AI AppSec Agent That Finds, Validates, and Fixes Vulnerabilities**

OpenAI launched **Codex Security** yesterday evening (March 6, 2026) in research preview for ChatGPT Enterprise, Business, and Edu customers. This is not a vulnerability scanner. This is an autonomous AppSec agent that ingests a codebase, builds a project-specific threat model, maps vulnerabilities with their root causes in context, validates findings to minimize false positives, and proposes targeted remediations aligned with the system's architecture and intent.

The capability set represents a qualitative jump beyond conventional SAST tooling: Codex Security runs full agentic reasoning loops rather than pattern matching against a rule set. It can distinguish between a vulnerability that is exploitable in the production deployment context versus one that is present in code but unreachable in practice — the false positive reduction problem that has plagued SAST tools for a decade. Bloomberg's headline framed it as potentially "cutting into demand for legacy cyber firms," and the Seeking Alpha note confirmed cybersecurity stocks showed mixed reactions on the news.

The dual-use architecture is unavoidable. The same reasoning engine that finds vulnerabilities defensively can find them offensively. Codex Security's research preview availability to Enterprise customers is broad enough that the methodology will be understood and replicated. What OpenAI is releasing defensively, threat actors will operationalize. The Anthropic/Mozilla Firefox disclosure from earlier this week — 22 CVEs from one automated analysis pass — is the template. Codex Security is that capability, productized and available to any Enterprise organization.

**The market disruption is also real:** OpenAI is offering free usage for the first month. The entry price point and capability ceiling are designed to displace Veracode, Checkmarx, and Snyk's core SAST/DAST market. The AppSec industry's 30-year run on rule-based analysis is encountering its GPT moment.

**Practitioner guidance:** Evaluate Codex Security in research preview if your organization has ChatGPT Enterprise access. Run it against your highest-risk repositories first — the ones that combine privileged access with complex input handling. Use the output as a second-opinion layer on top of existing tooling, not a replacement. The false positive reduction claim is worth testing empirically against your codebase before committing. More immediately: treat this launch as a deadline for any software your organization builds that hasn't had an AI-assisted security review. If OpenAI's agent can find bugs in your code, assume threat actors are running equivalent analysis.

### **2. GlicJack (CVE-2026-0628, CVSS 8.8) — Chrome's Gemini Live Panel Was a Privilege Escalator for Rogue Extensions**

Palo Alto Networks Unit 42 published the full technical write-up this week for **CVE-2026-0628**, dubbed **GlicJack** (Gemini Live In Chrome Jacking) — a high-severity vulnerability in Google Chrome where malicious browser extensions could inject JavaScript into the privileged Gemini Live side panel, escalating from basic extension permissions to system-level access. Google patched this in Chrome 143 (January 2026); the detailed methodology dropped this week.

The technical root cause: Chrome's Gemini Live panel runs in a privileged context due to its AI assistant capabilities — it has access to the local filesystem, camera/microphone APIs, and can screenshot any website. Chrome's WebView tag implementation contained "insufficient policy enforcement" (the NIST NVD description) that allowed a rogue extension with minimal permissions to inject scripts into that privileged context. The result: a malicious Chrome extension could silently access local files and directories, capture screenshots of any active tab, and activate camera and microphone inputs — all without the user receiving any additional permission prompt.

**The attack chain is sobering in its simplicity.** A user installs a Chrome extension that appears benign — a productivity tool, an ad blocker variant, a developer utility. The extension requests minimal, standard permissions. Invisibly, it injects into the Gemini Live panel context and gains access to every capability the AI assistant has on the machine. There is no prompt to the user. There is no visible indicator. The only requirement: Gemini Live is enabled in Chrome.

The Register's framing was direct: "Chrome AI panel became privilege escalator for extensions." That's the architectural lesson. When an AI assistant is embedded in a browser with privileged access to local data and device hardware, every extension that can touch it inherits an indirect path to those capabilities. The extension threat model just expanded to include AI panel privilege escalation as an attack class.

**Practitioner guidance:** Ensure Chrome is updated past version 143. Audit your organization's Chrome extension policy — any organization permitting user-installed extensions without review has exposure to this class. Maintain a Chrome extension allowlist for managed devices. For high-security environments: consider disabling the Gemini Live panel until you've audited extension inventory. The patched vulnerability is gone; the architectural lesson — AI panels with privileged access are extension attack surfaces — is permanent.

### **3. PleaseFix / PerplexedBrowser: Zenity Labs Discloses Zero-Click Comet Exploit That Drained 1Password Without Touching 1Password**

Zenity Labs published their full disclosure this week for **PleaseFix** and **PerplexedBrowser** — two critical vulnerability classes affecting agentic browsers, specifically demonstrated on **Perplexity Comet**. The disclosures represent the most thorough public analysis to date of how agentic browser trust models break in practice.

**PerplexedBrowser** (file:// path traversal): Comet's agentic capabilities — the ability to act on websites on behalf of the user — were not bounded against local filesystem paths. An attacker could trigger the agent to navigate to `file:///` paths, giving the AI agent full read access to the local machine's file system. The agent would exfiltrate files as part of a legitimate-appearing task, with no user prompt or warning. Perplexity issued an initial fix on January 23, 2026, blocking direct `file://` access. Zenity bypassed it using `view-source:file:///` — a trivially different URI scheme that achieved the same access. The confirmed-effective patch landed February 13.

**PleaseFix** (zero-click calendar invite to credential theft): The second exploit chain is architecturally more alarming. A malicious calendar invite, when processed by Comet's agentic email/calendar integration, triggered the agent to perform authorized actions within authenticated user sessions — including actions in 1Password's web interface. The agent, acting under legitimate user delegation, could access and exfiltrate vault credentials without the 1Password application itself being exploited. No vulnerability in 1Password. No vulnerability in the calendar application. Pure trust inheritance: the agent is authorized to act on your behalf, so when it accesses your password manager's web interface, it's doing exactly what it was designed to do.

Zenity's CTO stated: *"This is not a 1Password-specific vulnerability — it's an abuse of the security model of AI browsers, since this class of attack takes advantage of the agentic browser inherently acting on the user's behalf."* That framing is precise. The "PleaseFix" vulnerability class isn't patchable at the software layer. It requires architectural decisions about what actions agents can take in authenticated user sessions — decisions that most agentic browser deployments haven't made.

The calendar invite attack vector is the operational signal for enterprise security teams. Your users are receiving calendar invites from untrusted external sources every day. If those invites can trigger an agentic browser to perform authorized actions in authenticated sessions, you have a social engineering attack that produces actions rather than clicks.

**Practitioner guidance:** Update Perplexity Comet if you've deployed it. More broadly: audit every agentic browser in your enterprise environment for file system access boundaries — can the agent navigate to `file://` paths? Can it be triggered by untrusted calendar or email content? Implement explicit action scope constraints on agentic browsers: define what classes of action the agent is permitted to take in authenticated sessions, and require human confirmation for any action in privileged applications (password managers, email, financial tools). Treat agentic browser trust inheritance as a DLP boundary — any application the agent can access on behalf of the user is a data exfiltration vector if the agent is compromised.

### **1. vLLM CVE-2026-22778: Crafted Video URLs Enable RCE via Multimodal Model Input**

Security researchers disclosed **CVE-2026-22778** this week — a critical-severity vulnerability in **vLLM** (the widely-deployed open-source LLM serving framework) affecting versions prior to 0.14.1. The flaw: maliciously crafted video URLs passed as multimodal inputs can trigger remote code execution on the vLLM serving host.

The attack surface is the multimodal input processing pipeline. When vLLM processes video inputs, the URL is fetched and the content is decoded for the model. A crafted URL pointing to a malicious video file can exploit the video decoding layer to achieve host-level RCE. The serving host runs with whatever privileges the vLLM process uses — often high, given model serving requirements.

The security implication is broader than any single CVE: multimodal model deployments (vision, video, audio) expand the attack surface from text inputs to every media format the model can process. Each new input modality creates a new parsing layer. Each parsing layer has a vulnerability history. The combination of large attack surface, complex dependencies, and privileged execution environments makes multimodal LLM serving a rich target class.

Organizations running vLLM with multimodal capabilities that accept external URLs or user-supplied media inputs have an unpatched RCE unless they've updated to 0.14.1.

**Practitioner guidance:** Update vLLM to 0.14.1 immediately if you're running any version below this with multimodal input processing. If user-supplied video or image URLs are accepted by your serving infrastructure, implement URL allowlisting and content-type validation before the input reaches the model. Treat multimodal model serving as a distinct attack surface from text-only deployments — the media processing layer requires its own security review. Run Trivy against your vLLM deployment to catch dependency chain vulnerabilities.

### **2. Unit 42 Web Prompt Injection Research: Confirmed Wild Exploitation of AI Agents via Malicious Sites**

Palo Alto Networks Unit 42 published research this week confirming **web-based indirect prompt injection against AI agents in production** — not in a research lab, in the wild. The attack: a user's AI agent browses to a website containing adversarially crafted content. The content is invisible to the human user but processed by the AI agent as instruction. The agent executes the embedded instructions as part of normal task processing.

The Unit 42 finding confirms what OpenAI has acknowledged in the Atlas browser context: indirect prompt injection through web content is an active exploitation technique, not a theoretical concern. The attack doesn't require compromising the target website at the infrastructure level — injecting adversarial content into any content the AI agent processes (product descriptions, blog posts, web pages the agent summarizes) is sufficient.

The sampling attack variant — documented in the research as a distinct technique — exploits the probabilistic nature of LLM outputs. By crafting inputs that bias the probability distribution of model responses toward attacker-preferred outputs, rather than including explicit injection instructions, sampling attacks reduce the detectability of the attack. The model's output looks plausibly natural rather than like executed code.

Combined with the Scout feed's independent confirmation of RAG pipeline poisoning achieving 90%+ manipulation success rates, the signal is consistent: any AI agent that ingests untrusted content as part of its context is operating in an adversarial content environment where significant portions of what it processes may be designed to manipulate its outputs or actions.

**Practitioner guidance:** If you operate AI agents that browse the web, summarize documents, or ingest external content: implement a content validation layer that runs before that content enters the agent's context. Treat web content as untrusted input with the same rigor as SQL inputs — injection defenses apply. Consider sandboxing agents that browse arbitrary web content from agents that take consequential actions: a research agent that summarizes pages should not also have access to email, file system writes, or API calls.

### **3. Sarvam 105B: India's First Competitive Open-Source LLM — and What Open-Weight Models Mean for the Security Equation**

Sarvam AI released **Sarvam 105B** this week — India's first competitive large-scale open-source language model, trending on HN this morning with 92 points and 25 comments. The model targets Indian language capabilities (22 official Indian languages) alongside English, positioning as a sovereign AI alternative to Western frontier models.

The AI security relevance is architectural, not India-specific: every competitively capable open-weight model that ships is a model that can be downloaded, fine-tuned, and deployed without vendor safety controls, usage monitoring, or rate limiting. Sarvam 105B joins Llama, Mistral, Qwen, and DeepSeek as a capable model available for unconstrained deployment — including offensive use cases.

The security community's open-weight model concern is precise: fine-tuning a capable open-weight model to bypass alignment constraints requires significantly less compute than training from scratch. Each new open-weight model at the 70B-105B capability tier is a new base for jailbreak fine-tunes, malware generation models, and automated vulnerability discovery agents that operate without the safety controls of the hosted alternatives. The proliferation of capable open-weight models is the infrastructure layer for the next generation of AI-assisted offensive tooling.

The flip side, documented in the newsletter over the past week: the same open availability enables defensive use cases. Researchers who found Clinejection, built the MCP security hardening playbooks, and developed mcp-scan are using accessible models. The advantage asymmetry runs both ways.

**Practitioner guidance:** Update your threat model to account for unconstrained deployment of 70B-105B parameter models. Threat actors with modest GPU resources can fine-tune these models for offensive purposes. Security controls that depend on API-level rate limiting or vendor safety filters are insufficient against locally-deployed open-weight models. The offensive capability level you should assume from sophisticated threat actors is now effectively frontier-model-equivalent for narrow tasks.

The class of attack — extension-to-AI-panel-to-host-privilege escalation — is now documented, understood, and available for threat actors to adapt against any future AI browser feature.

3. **PleaseFix's credential theft chain requires architectural controls, not software patches.** The Perplexity Comet calendar-invite-to-1Password attack worked because the agent was authorized to act in authenticated sessions. Patching the file:// boundary doesn't eliminate the trust inheritance model. Audit every agentic browser in your enterprise: define explicitly what categories of action agents are permitted to take, implement human-in-the-loop confirmation for privileged application access (password managers, financial tools, email), and treat agentic browsers as DLP boundary devices.

4. **Your RAG knowledge base is now your SQL injection surface.** Scout's top signal today: RAG pipeline poisoning achieving 90%+ manipulation success rates without query-level injection. Any document in your RAG datastore that was ingested from an external, untrusted source is a potential adversarial influence vector. Implement access control on vector DB write paths. Audit existing datastores for unexpected content. Do not allow untrusted external sources to write directly to production RAG datastores.

5. **vLLM CVE-2026-22778 is a patch-now story.** If you run vLLM with multimodal input processing and accept user-supplied video or image URLs, you have an unpatched RCE path if you haven't updated to 0.14.1. Patch today. Then run a multimodal input security review: every media format your model accepts is a new parsing layer with its own vulnerability history. Treat multimodal serving with the same attack surface rigor as a web server parsing untrusted file uploads.
